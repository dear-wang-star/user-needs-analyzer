# Implementation Plan

## 1. Skill 目标与边界

### 目标
- 将分散的用户反馈材料整理为结构化证据。
- 基于证据提炼、校验并输出用户需求，而不是直接给出功能方案。
- 为后续 Skill 实现准备稳定的输入校验、分析模板和测试样例。

### 边界
- V1.0 只处理“用户需求分析”骨架，不覆盖产品路线图生成、自动方案设计、工单分类系统或可视化仪表盘。
- V1.0 不引入外部依赖安装，不接入在线服务，不扩展到多仓库联动。
- 本阶段不编写完整方法论文档，只搭建文件结构、占位文件和施工计划。

## 2. V1.0 输入与输出

### 输入
- 单条或多条用户反馈文本。
- 访谈摘要、客服工单、功能请求等原始材料。
- 可选的上下文信息，如产品模块、用户类型、时间范围。

### 输出
- 结构化证据表。
- 候选用户需求卡片。
- 需求优先级排序结果。
- 最终分析报告草稿。

## 3. 分阶段施工步骤

### Phase 0: 工程骨架
- 创建仓库目录、占位文件和实施计划。
- 补充 README 中的建设中说明和目录结构。

### Phase 1: Skill 外层协议
- 编写精简版 `SKILL.md`，只定义任务目标、输入输出和调用规则。
- 编写 `AGENTS.md`，说明协作约束、禁区和验收口径。

### Phase 2: 参考规则沉淀
- 逐步填充 `references/` 下的方法参考文件。
- 明确“证据”“需求”“方案”“优先级”的判断标准。

### Phase 3: 模板落地
- 完成 `templates/` 中的结构化输出模板。
- 统一字段名、证据引用方式和结论格式。

### Phase 4: 校验脚本
- 为输入完整性、证据引用一致性、最终报告格式编写基础校验脚本。
- 保持脚本轻量、可本地运行、无额外依赖。

### Phase 5: 示例与测试
- 在 `examples/` 中准备三类典型样本。
- 在 `tests/` 中补齐正常、证据不足、误导性样例。

## 4. 文件职责

### 根目录
- `SKILL.md`: Skill 主入口说明，后续定义行为边界和执行流程。
- `AGENTS.md`: 协作说明、角色边界、施工约束。
- `.gitignore`: 后续忽略本地缓存、临时输出和测试产物。
- `IMPLEMENTATION_PLAN.md`: 当前施工计划和阶段拆解。
- `README.md`: 仓库说明与目录导航。

### references/
- `evidence-taxonomy.md`: 定义证据类型与分类维度。
- `need-vs-solution.md`: 区分用户需求与方案表述。
- `requirement-validation.md`: 需求成立条件与校验规则。
- `evidence-strength.md`: 证据强弱与可信度判断。
- `prioritization-rules.md`: 需求排序与优先级规则。

### templates/
- `evidence-table.md`: 证据整理表模板。
- `user-need-card.md`: 单条用户需求卡模板。
- `needs-ranking.md`: 需求排序模板。
- `final-report.md`: 最终报告模板。

### scripts/
- `validate_input.py`: 校验输入材料结构与必要字段。
- `validate_evidence_links.py`: 校验证据引用与交叉链接。
- `validate_report.py`: 校验最终报告格式和核心段落。

### examples/
- `single-feature-request/`: 单一功能诉求样例。
- `interview-synthesis/`: 访谈归纳样例。
- `support-ticket-analysis/`: 工单聚类分析样例。

### tests/
- `normal-cases.json`: 正常输入与预期结果样例。
- `insufficient-evidence-cases.json`: 证据不足场景样例。
- `trap-cases.json`: 容易把方案误判为需求的陷阱样例。

## 5. 测试与验收方式

### 结构验收
- 目录和文件完整存在。
- 空目录通过 `.gitkeep` 可被 Git 跟踪。
- `README.md` 与实际目录结构一致。

### 内容验收
- `SKILL.md` 后续应保持精简，不写成长提示词。
- `references/` 只沉淀规则，不堆砌营销式描述。
- `templates/` 字段命名统一、可复用。

### 脚本验收
- 校验脚本可独立运行。
- 对缺字段、断链证据、报告缺段落等情况能给出明确失败信号。

## 6. 可能风险

- 过早扩大范围，把“需求分析”做成“解决方案生成”。
- `SKILL.md` 写得过长，变成难维护的提示词文档。
- 参考规则与模板字段不一致，导致后续脚本难以校验。
- 测试样例不足，无法覆盖“证据不足”与“伪需求”场景。
- 过度依赖脚本格式校验，忽略分析结论本身的质量判断。
